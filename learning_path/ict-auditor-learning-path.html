ICT Auditor Learning Path (Draft)

Accessibility Auditor

After completing the learning path, you will have a basic understanding of the technologies that the Web is built on, a basic understanding of accessibility as it pertains to the Web, mobile, documents, and eventually desktop; and how to build and evaluate rich internet applications. No course can be exhaustive. Most knowledge will be gained through experience. This learning path will lay the foundation allowing you to build upon it as you practice web development and accessibility auditing.

Most training courses in this learning path will either be free, or readily paid for. (Courses are free unless a cost is listed.) Most are self-paced and start as soon as you want so you can start learning without delay.

Note: the courses provided by the Government of Canada are available in both Official Languages. The rest, however, are mostly English only.

Target Audience:

  • ICT Accessibility Auditors
  • Those interested in web development with an eye towards accessibility

Block B - Introduction to Web Fundamentals

Objective: You will learn the fundamental building blocks of a website: HTML, CSS, JavaScript, and build a simple website hosted on GitHub pages.

What does this have to do with accessibility?

It's said that the best way to identify a counterfeit dollar bill is to know what a real dollar bill looks like. Likewise, the best way to spot bad HTML is to know proper HTML. HTML was designed from the beginning to be accessible to users of assistive technologies.

HTML is a semantic language, meaning that its elements have meaning. Certain HTML elements expose themselves to assistive technologies in special ways. For example, an <h1> may look big and bold, but it means that it's a top-level heading. The difference between regular text styled to look big and bold is completely different to a screen reader user. It is important to understand the meaning behind the different HTML elements and attributes.

What does Git Have To With Accessibility

Nowadays, in a technical field such as web development, knowing Git is a bit like knowing about MS Excel Pivot Tables in data analysis. You can probably get by for a while without knowing it. But sooner or later you're going to need to learn it. You don't need to become an expert. But when time comes to use it, you don't want to be starting from scratch.

You'll start by building a small website. GitHub allows websites to be hosted on their GH Pages. So before doing that, we'll start with learning basic Git.

Block C - Web Accessibility

Objective: To learn the fundamental principles of accessibility, WCAG (Web Content Accessibility Guidelines) 2.1 at the AA level, and performing accessibility audits.

Now that you have a foundation of web development, it's time to build on that. We now turn our attention to accessibility and WCAG.

The term "Accessible" could be defined as "usable by all". Whether or not something, say a website, is accessible is somewhat subjective. Having a standard makes it objective. WCAG 2.1 (Level AA) is the standard we use. As an example, how much contrast between adjacent colours is needed varies by user. WCAG gives a definitive answer: for text against background, it's 4.5 to 1.

It's important to understand basic accessibility principles like different disability groups and how they each interact with the Web. And it's important to know WCAG so you can interpret, apply, and reference it during an accessibility audit.

As for which specific courses to take, there is some choice. Ideally, you should take the T716 course offered by Canada School of the Public Service. But spots may not be readily available. Alternatively, the Accessibility for Web Development course offered by ESDC College is offered more frequently. But it's not as in-depth, and it can still be a few weeks before you can get a spot. And you want to start learning now.

Below are some options to get you started today.

Block E - Mobile Accessibility

Objective: To learn the fundamental principles of accessibility as it applies to Mobile devices and apps.

The most important part of learning mobile accessibility is learning to use a screen reader on a mobile device - without a keyboard. On desktops, keyboards are part of the screen reader experience. You need to learn shortcuts and hotkeys. You need to be able to use the computer without a mouse plus the screen reader shortcuts. But for mobile, you need to learn to use a screen reader using only your fingers. While mobile devices can have keyboards connected, finger-only usage would be considered the primary way to interact.

Block F - Advanced Topics

WAI-ARIA

Now that you have a foundation of web development and accessibility, it's time to build on that. We now turn our attention to WAI-ARIA, or "ARIA" for short.

ARIA is used to enhance accessibility in web applications. You will need to be able to identify improper ARIA in web audits, and offer advice on how to fix errors.

Learning WAI-ARIA is a years-long persuit, learning what roles require which parent and child roles, which stand on their own, what browser and screen reader combos support what aria, etc. But you will run into WAI-ARIA sooner rather than later.
