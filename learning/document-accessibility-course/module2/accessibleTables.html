What are accessible tables?

Accessible tables organize data in a way that everyone can understand, including people using screen readers, voice control, and other assistive technologies. Well-structured tables provide clear relationships between data, making information easier to find and compare across different platforms and applications.

Unlike web tables that use HTML markup, document tables rely on built-in accessibility features in applications like Microsoft Word, Excel, and PowerPoint. These features help assistive technologies understand table structure and navigate data efficiently.

When and why to use tables

Use tables when you need to:

  • Compare data: Show relationships between different categories, such as budget items by department
  • Display structured information: Present systematic data like schedules, contact lists, or specifications
  • Organize complex data: Make large datasets easier to scan and understand
  • Show numerical relationships: Compare values, percentages, or trends across categories

Don't use tables for:

  • Layout purposes: Use proper formatting tools instead of invisible tables for positioning
  • Simple lists: Use bullet points or numbered lists for single-column information
  • Text alignment: Use paragraph formatting rather than table cells
  • Visual spacing: Use proper margins and spacing tools instead of empty table cells

Tables with charts and graphs

Tables and charts work together to improve accessibility. While charts provide visual insight, tables offer the precise data that assistive technology users need. This combination ensures everyone can access and understand your information.

How to combine tables and charts effectively:

  • Provide the data table: Include the underlying data that your chart represents
  • Place tables strategically: Put tables immediately after charts or in an appendix with clear references
  • Use consistent terminology: Match column headers and labels between charts and tables
  • Summarize key insights: Add a brief text summary of the main trends or findings

Best practices for accessible tables

Essential accessibility principles:

  • Keep it simple: Use the simplest table structure that effectively presents your data
  • Logical reading order: Organize data so it makes sense when read left-to-right, top-to-bottom
  • Consistent formatting: Use the same format for similar data types throughout your document
  • Clear relationships: Make sure column headers clearly describe the data below them
  • Avoid merged cells: Merged cells confuse assistive technologies and should be used sparingly

Platform-agnostic principles:

  • Use built-in table tools: Every application has proper table creation tools—use them instead of manual formatting
  • Enable table accessibility features: Turn on header row designation and other accessibility options
  • Test with navigation: Try using Tab and arrow keys to move through your table
  • Consider export formats: Ensure your table structure survives conversion to PDF or other formats

Clear headers and captions

Writing effective table captions:

  • Be descriptive: Explain what data the table contains and its scope
  • Include timeframe: Specify dates, quarters, or other time periods when relevant
  • Mention units: Include currency, percentages, or measurement units
  • Keep it concise: One clear sentence is usually sufficient

Designing clear headers:

  • Use descriptive labels: Avoid abbreviations or unclear terms
  • Be specific: "Total Sales ($)" is better than just "Total"
  • Use consistent terminology: Don't switch between "Revenue" and "Sales" for the same concept
  • Consider context: Headers should make sense even when read independently

Complex tables and relationships

Sometimes you need tables with multiple header levels or complex relationships. While these can be accessible, simpler alternatives are often better for everyone.

Strategies for complex data:

  • Split complex tables: Break one complex table into several simpler ones
  • Use consistent patterns: Establish clear patterns for how data relates across rows and columns
  • Add explanatory text: Include a paragraph explaining how to read the table
  • Consider alternative formats: Sometimes lists, charts, or summary paragraphs work better

When you must use complex tables:

  • Group related headers: Use consistent formatting to show header relationships
  • Provide reading instructions: Explain the table structure before presenting the data
  • Test thoroughly: Have colleagues review the table using only keyboard navigation
  • Consider summaries: Provide key findings in text form as well

  • Progressive disclosure: Start with overview data, then provide detailed views

How to create accessible tables step-by-step

Follow these platform-agnostic steps to create accessible tables in any application:

Step 1: Plan your table structure
  1. Identify your data: What information needs to be organized?
  2. Determine relationships: How do different data points relate to each other?
  3. Choose the simplest structure: Use the minimum number of columns and rows needed
  4. Plan your headers: What labels will make each column and row clear?
  5. Write your caption: How will you describe the table's purpose?
Step 2: Create the table using proper tools

In Microsoft Word, Excel, or PowerPoint:

  1. Go to Insert → Table
  2. Choose the number of rows and columns
  3. Use the Table Design tab to designate header row
  4. Add a table title using the table properties

In other applications:

  1. Look for Table or Insert Table functions
  2. Avoid manual formatting with tabs or spaces
  3. Use built-in table accessibility features when available
Step 3: Add content and formatting
  1. Add your caption: Place a descriptive title above or within the table
  2. Fill in headers: Use clear, descriptive labels for all columns and rows
  3. Enter data consistently: Use the same format for similar data types
  4. Apply minimal formatting: Use simple, consistent styling that doesn't interfere with structure
  5. Avoid merged cells: Keep the structure simple whenever possible
Step 4: Enable accessibility features

Check these settings in your application:

  • Header row designation: Ensure the first row is marked as headers
  • Header column designation: Mark the first column as headers if applicable
  • Table properties: Add alt text or description if your application supports it
  • Reading order: Verify the table reads logically from left to right, top to bottom

Note: Table alt text support varies across assistive technologies in Office documents; prioritize captions, table headers, and nearby narrative as primary mechanisms.

Application-specific features:

  • Microsoft 365: Use Table Design → Header Row and Table Properties → Alt Text
  • PDF creation: Ensure table structure is preserved when exporting
Step 5: Test your table
  1. Navigation test: Use Tab and arrow keys to move through the table
  2. Screen reader test: If available, test with screen reader software
  3. Reading order test: Read the table aloud to check if it makes sense
  4. Context test: Can someone understand the table without seeing surrounding text?
  5. Export test: Check if the table structure survives conversion to PDF or other formats

Check your work

Quick accessibility checklist:

  • Table has a clear caption or title
  • Headers are designated and descriptive
  • Data is organized logically
  • Reading order makes sense (left-to-right, top-to-bottom)
  • Consistent formatting within columns
  • Minimal use of merged cells
  • Table can be navigated with keyboard only
  • Structure is preserved when exported

User testing questions:

  • Can someone understand the table purpose from the caption alone?
  • Are the headers clear without seeing the data?
  • Does the table make sense when read row by row?
  • Can users find specific information quickly?
  • Would this table be useful if converted to audio?
Common issues and fixes
Troubleshooting accessible table problems
Problem Solution
Headers not recognized by assistive technology Use your application's header designation feature, not just bold formatting
Table structure lost when exported to PDF Check export settings for accessibility options; avoid image-based PDFs
Complex table is confusing to navigate Split into multiple simpler tables or provide alternative presentation formats
Data relationships are unclear Add explanatory text before the table and improve header specificity
